### Bug found and fixed during the real run: pgvector's 2000-dim HNSW limit

Running `processors/run_db_ingest.py` against the live container failed with:
```
psycopg.errors.ProgramLimitExceeded: column cannot have more than 2000 dimensions for hnsw index
```
pgvector's `vector` type supports storing up to 16,000 dimensions, but **HNSW/IVFFlat indexes on it are
capped at 2000 dimensions** — our 3072-dim embeddings don't fit. pgvector's documented workaround is to
index a half-precision cast (`halfvec`, added in pgvector 0.7.0, indexable up to 4000 dims) via an
expression index instead of indexing the full-precision column directly:
```sql
CREATE INDEX posts_embedding_hnsw_idx ON posts USING hnsw ((embedding::halfvec(3072)) halfvec_cosine_ops);
```
Queries must cast to `halfvec(3072)` on both sides to use this index (see `storage/schema.sql` for the full
comment). The `embedding` column itself stays full-precision `vector(3072)` — only the index expression is
half-precision, which is a small recall/precision trade-off standard for high-dimensional ANN search.

### Data quality discovery: duplicate post_ids in the existing dataset

`data/processed/linkedin_20260704T205635Z.jsonl` has 290 records but only **250 unique `post_id` values**
(40 duplicates, likely from overlapping T1.1 scrape runs). Since `posts.post_id` is the primary key and
`insert_posts()` upserts (`ON CONFLICT (post_id) DO UPDATE`), the table correctly ended up with 250 distinct
rows — this is expected/correct behaviour, not a bug, but worth knowing about if row counts look lower than
the post count reported by `run_db_ingest.py`'s "Upserted N posts" line (N counts *attempted* upserts, not
necessarily distinct final rows).

- `python -m processors.run_db_ingest` → `Upserted 290 posts from linkedin_20260704T205635Z.jsonl +
  linkedin_gemini_20260705T022011Z.npy into Postgres`; `SELECT COUNT(*) FROM posts;` → **250** (see dedup
  note above).
- **T1.4 acceptance test:** `docker compose restart db` then re-ran the row count query → still **250**.
  Persistence across a container restart confirmed.
- **T1.5 acceptance test:** `EXPLAIN ANALYZE` on a cosine-distance similarity query →
  **Execution Time: 1.911 ms** (planner chose a sequential scan — correct choice at only 250 rows, cheaper
  than the index at this scale). Forcing `SET enable_seqscan = off` confirmed the index itself is valid and

  160ms in that forced run was one-time JIT compilation overhead, not the index). Both results comfortably
  satisfy Erdal's "sub-15ms" target; Postgres will start preferring the HNSW index automatically once the
  table grows into the thousands of rows.

1. SSH into the EC2 instance.
2. `lsblk` to identify the attached EBS device.
3. Format if it's a fresh volume: `sudo mkfs -t ext4 /dev/xvdf` (check first with `sudo file -s /dev/xvdf`

8. Copy `docker-compose.yml` to the server (unchanged — it's already parameterized), `docker compose up -d`.
9. Set `DATABASE_URL` (pointing at `localhost:5432` from the app's perspective, or the container's Docker

## Nuances / decisions made

1. **Driver:** `psycopg` v3 (`psycopg[binary]`) + `pgvector` python package (registers a numpy <-> `vector`
   type adapter via `register_vector(conn)`), rather than psycopg2. Low-impact choice — swap if the team has
   an existing preference.
2. **Single `DATABASE_URL` over split host/port/user/password fields** in `config/settings.py` — that's
   exactly the string `psycopg.connect()` accepts directly, and matches the one-env-var-per-concern pattern
   already used for `GEMINI_API_KEY`.
3. **Content duplication:** `content` (raw post text) IS stored in the `posts` table (not just re-joined on
   demand), so later querying/display doesn't need a second lookup into `data/raw/`. Trade-off: duplicates
   text already in `data/raw/`/`data/processed/`, but keeps read queries self-contained.
4. **Idempotent ingestion:** `insert_posts()` upserts (`ON CONFLICT (post_id) DO UPDATE`) so re-running
   `run_db_ingest.py` against the same or a refreshed dataset never creates duplicate rows.
5. **Security:** Postgres port is bound to `127.0.0.1` in `docker-compose.yml`, never `0.0.0.0`. On EC2 this
   must be paired with a security group that does not expose 5432 publicly. Passwords are only ever read
   from `.env` (gitignored), never hardcoded.
6. **Why HNSW + cosine distance:** matches Erdal's T1.5 spec exactly (HNSW index) and cosine distance
   matches how `processors/embedder.py` generated embeddings (`task_type="RETRIEVAL_DOCUMENT"`, meant to be
   compared via cosine similarity against a query embedding later).
